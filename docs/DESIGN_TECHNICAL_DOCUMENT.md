## 1. 比赛准备和调研

### 1.1 项目目标描述

SafeOps Agent 的目标是设计并实现一套面向国产麒麟操作系统的安全智能运维 Agent。系统接收运维人员的自然语言指令后，先理解其运维意图，再在多层安全护栏的约束下映射到最小权限工具执行，全程记录可回放的思维链与防篡改审计日志，最终实现"高效、可解释、可追溯、可杜绝误操作"的对话式运维。

项目目标可以拆成六项：

1. 构建自然语言运维入口：运维人员用中文自然语言描述诉求，系统理解意图并选择合适的运维动作，而不要求记忆命令与参数。
2. 建立多层安全护栏：在输入侧筛查高危意图，在输出侧过滤大模型文本，在执行前按风险等级裁决，杜绝任意 shell 执行与越权操作。
3. 实现三级风险策略：低风险只读操作直接执行，中风险变更操作需一次性确认令牌，高风险破坏性操作默认拒绝。
4. 构建最小权限工具集：把运维能力封装成带风险标注的白名单工具，工具之外无执行路径。
5. 生成可追溯审计：把每次决策拆成五步思维链，并以哈希链方式落盘，做到"改一条即可被发现"。
6. 完成可演示交付：提供命令行、Web 工作台、标准 MCP 协议服务端、操作手册和设计文档，满足初赛开源仓库审查和视频演示要求。

### 1.2 赛题理解与运维场景

赛题的核心不是再造一个运维命令行工具，而是构建一个能理解自然语言、能安全地与操作系统交互、并能对每一步决策负责的智能运维 Agent。对使用者和评委而言，系统需要回答三个问题：

- 用户说了一句自然语言，系统打算执行什么，依据是什么；
- 这个动作有多大风险，凭什么允许、拒绝或要求确认；
- 事后如何证明系统确实按规则执行，且记录没有被篡改。

因此，本项目把"安全护栏优先、决策可解释、操作可追溯"作为设计底线。SafeOps Agent 不追求"能执行任意命令"，而是在明确的权限边界内，把意图理解、风险裁决、工具执行和审计记录串成一条可复核的安全链路。

### 1.3 安全运维与相关技术调研

项目调研了三类现实运维痛点：一是手工运维依赖命令记忆、易误操作；二是直接把大模型接到 shell 会带来不可控的执行风险；三是运维审计常常只有"结果日志"，无法自证未被删改。这些痛点共同指向一个结论：智能运维系统的价值不只在"听懂人话"，更在"安全地执行"和"可信地记录"。

围绕麒麟/Linux 目标环境，项目确认了核心系统能力的落地方式：系统信息来自 `os-release` 与 `uname`，服务生命周期通过 `systemctl` 管理，日志通过 `journalctl` 读取，监听端口通过 `ss` 采集，软件包通过 `rpm` 查询。开发阶段主要在 Windows 上进行，因此这些命令在 Windows 上返回兼容说明文本，在麒麟/Linux 上真实执行，并整理了实机验证清单待补测。

### 1.4 相关技术调研与路线取舍

围绕赛题，本项目重点调研并取舍了四类技术路线：

| 方向 | 可用能力 | 取舍结果 |
| --- | --- | --- |
| 直接调用 shell | 实现简单，能力最全 | 风险不可控，明确放弃，改为最小权限白名单工具 |
| 关键词规则意图识别 | 离线、确定、零成本 | 作为默认与回退路径保留，保证无网络也能运维 |
| 大模型意图理解 | 语义泛化强，能追问参数 | 只作可选增强，必须受输入/输出护栏、超时和失败回退约束 |
| MCP 协议接入 | 标准化工具调用，可被通用客户端复用 | 采用标准 JSON-RPC 2.0 stdio 服务端，与内部安全策略共用一套裁决 |

最终路线是"自然语言输入 + 大模型/规则双通道意图理解 + 三级风险策略 + 最小权限工具执行 + 哈希链审计 + 标准 MCP 协议"。这条路线兼顾可运行、可解释、可演示和安全可控，适合初赛阶段的安全智能运维场景。

## 2. 系统框架和模块设计

### 2.1 总体架构

SafeOps Agent 按分层方式设计，一次自然语言运维请求自上而下经过六个阶段，每一阶段都保留可审计的中间结论。整体数据流如下：

```
自然语言运维请求（CLI / Web / MCP 客户端）
        |
        v
入口与会话调度层：cli.py / web_server.py / mcp_stdio.py，统一进入 Agent
        |
        v
意图理解与工具选择层：agent.py + llm/ ，大模型意图理解，失败回退规则匹配
        |
        v
安全策略与风险裁决层：security/policy.py ，输入护栏 + 参数校验 + 风险分级
        |
        v
工具执行与受管变更层：tools/ ，24 个最小权限白名单工具，写前快照可回滚
        |
        v
审计与协议服务层：audit/logger.py ，五步思维链 + 哈希链防篡改 + 校验
        |
        v
响应返回：结果摘要 / 确认令牌 / 拒绝原因 + 完整思维链
```

从工程视角看，这个架构的核心不是单个模型调用，而是一条"意图—风险—执行—审计"的可复核安全流水线。入口层接收自然语言；中间层把文本逐步转换为 `PolicyDecision`、`ToolResult`、思维链步骤等结构化对象；输出端再基于这些对象生成响应，并把完整决策过程写入哈希链审计日志。每一层都保留中间结论，既便于定位问题，也便于在演示时解释"系统为什么做出这个决定"。

分层设计还解决了两个运维评审场景中的现实问题。第一，评委需要看到"为什么允许/拒绝"，而不只是执行结果，因此风险裁决层不能被意图理解层跳过。第二，安全规则、工具集和意图理解通道都会持续演进，如果全部耦合在一个脚本里，任何一处调整都会威胁主流程稳定性。SafeOps Agent 把意图理解、风险裁决、工具执行和审计记录拆开，使每一层都能独立测试、独立回退。

系统主流程可以概括为"先守护栏，再动执行"。事实与判断由本地策略完成，包括意图筛查、风险等级、参数合法性和确认状态；语义理解可选地由大模型增强。这样即使大模型不可用，系统仍能基于规则完成意图识别、风险裁决和工具执行；即使启用大模型，它也只能在护栏内影响"选哪个工具"，而不能突破策略直接执行动作。

### 2.2 入口与会话调度层

入口层由 `cli.py`、`web_server.py` 和 `mcp_stdio.py` 三个入口组成，分别面向命令行、浏览器和标准 MCP 客户端，最终都汇聚到同一个 `SafeOpsAgent`，共享同一套安全策略与审计逻辑。

CLI 的参数设计尽量显式、无隐式副作用：`--yes` 表示确认执行中风险操作，`--confirm <令牌>` 表示凭一次性令牌精确执行已预演动作，`--verify-audit` 校验审计哈希链完整性，`--list-tools` 输出 MCP 风格工具清单，`--json` 输出完整结构化结果。

代码印证：`src/safeops_agent/cli.py` 把令牌确认与普通请求分成两条明确路径，避免把"确认执行"和"重新理解意图"混在一起。

```python
if args.confirm:
    response = agent.confirm(args.confirm)
else:
    response = agent.handle(args.request, confirmed=args.yes)
```

入口层还负责稳定的演示体验。例如 Windows 控制台默认 GBK 编码会把中文输出打成乱码，CLI 在启动时统一把标准输出切换到 UTF-8；Web 端则按会话隔离上下文，使慢请求互不阻塞。这些细节降低了演示视频录制和现场答辩的不确定性。

### 2.3 意图理解与工具选择层

意图理解由 `agent.py` 的 `_smart_select_tool` 统一调度，采用"大模型优先、规则兜底"的双通道设计：先尝试大模型意图理解，若大模型不可用、返回未知工具或解析失败，则自动回退规则关键词匹配，并把回退原因写入思维链，使降级可见而非静默发生。

代码印证：`src/safeops_agent/agent.py` 的工具选择逻辑在大模型命中合法工具时采用其结果，否则回退规则匹配并保留 `fallback_note`。

```python
if tool_name and tool_name in self.tools:
    return tool_name, args, "llm", reasoning, None, ""
if reasoning:
    fallback_note = reasoning

tool_name, args = self._select_tool(text)
return tool_name, args, "rule", "", None, fallback_note
```

大模型通道由 `llm/provider.py` 的 `DeepSeekProvider` 实现，采用 OpenAI 兼容接口，要求模型以 JSON 对象返回 `tool`、`args`、`reasoning` 和可选的 `clarification`。当模型识别到工具但参数不足时，可通过 `clarification` 向用户追问，而不是猜测参数。规则通道由 `RuleBasedProvider` 兜底，完全离线、不调用任何外部 API。

这一层的设计原则是"语义可以增强，安全不可让渡"。大模型只影响"选哪个工具、填哪些参数"，其输出仍要经过后续策略裁决；即使模型给出一个工具名，也必须命中白名单且通过参数校验才能执行。

### 2.4 安全策略与风险裁决层

安全策略由 `security/policy.py` 的 `PolicyEngine` 集中实现，是整个系统的护栏核心，包含三道关卡：输入侧意图筛查、工具参数校验、工具风险分级。

第一道关卡是意图筛查（`evaluate_intent`），在意图理解之前拦截破坏性关键词与"危险动作 + 敏感路径"组合。关键词覆盖 `rm -rf`、`mkfs`、`shutdown`、`删除根目录`、`格式化`、`关闭防火墙` 等中英文表述；对纯 ASCII 关键词采用单词边界匹配，避免误伤正常词汇。

第二道关卡是参数校验（`_validate_tool_args`），拦截命令注入字符与敏感路径，并对服务名、软件包名做安全标识符校验。

代码印证：`src/safeops_agent/security/policy.py` 对参数中的注入字符集统一拦截。

```python
argument_forbidden_chars = set(";&|`$<>") | {"\n", "\r", "\t", "\x00"}
...
if any(char in text for char in self.argument_forbidden_chars):
    return PolicyDecision(
        allowed=False,
        risk=RiskLevel.HIGH,
        reason=f"参数 {key} 包含命令注入风险字符",
        error_code="ARG_COMMAND_INJECTION",
    )
```

第三道关卡是风险分级（`evaluate_tool`），把工具风险落到三级策略：低风险直接放行，中风险要求确认，高风险默认拒绝。

```python
if tool.risk == RiskLevel.LOW:
    return PolicyDecision(allowed=True, risk=tool.risk, reason="只读低风险工具")
if tool.risk == RiskLevel.MEDIUM:
    if confirmed:
        return PolicyDecision(allowed=True, risk=tool.risk, reason="中风险工具已确认")
    return PolicyDecision(allowed=False, risk=tool.risk, reason="中风险工具需要用户确认",
                          requires_confirmation=True, error_code="TOOL_CONFIRMATION_REQUIRED")
return PolicyDecision(allowed=False, risk=tool.risk, reason="高风险工具默认拒绝", error_code="TOOL_HIGH_RISK")
```

三级风险策略的价值在于把"能不能做"从二元判断细化为三态处置。只读查询不打扰用户，破坏性操作直接挡住，真正需要人参与的是"有副作用但合理"的中风险变更——系统对这类操作签发一次性确认令牌，把最终决定权交回给人。

### 2.5 工具执行与受管变更层

工具集由 `tools/registry.py` 统一注册，当前提供 24 个带风险标注的白名单工具（19 个只读低风险 + 5 个中风险变更），覆盖系统信息、资源指标、进程、日志、服务状态、网络连接、监听端口、磁盘分区、用户、定时任务、软件包查询和多场景诊断（低风险），以及服务 start/stop/restart 和受管文件 apply/rollback（中风险）。工具之外没有任何执行路径，系统不提供"执行任意命令"的能力。

代码印证：`src/safeops_agent/tools/registry.py` 用 `ToolSpec` 声明每个工具的名称、风险等级、参数与分类，风险等级直接参与策略裁决。

```python
ToolSpec(name="system.resources", description="采集 CPU、内存和磁盘基础指标",
         risk=RiskLevel.LOW, handler=get_resource_usage, category="system"),
...
ToolSpec(name="service.restart", description="重启指定 systemd 服务",
         risk=RiskLevel.MEDIUM, handler=restart_service,
         parameters={"service": {"type": "string"}}, required=["service"], category="service"),
```

服务生命周期在麒麟/Linux 上通过 `systemctl` 真实执行，在 Windows 开发环境返回预告文本；受管文件写入被限制在项目内的 `data/managed/` 工作区，写入前自动打快照，回滚是可真正执行的逆操作而非仅给建议。

代码印证：`src/safeops_agent/tools/operations.py` 的 `_safe_name` 禁止路径穿越，把变更严格约束在受管工作区内。

```python
def _safe_name(name: str) -> str | None:
    name = name.strip()
    if not name or "/" in name or "\\" in name or ".." in name:
        return None
    allowed = set("abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789._-")
    if not all(char in allowed for char in name):
        return None
    return name
```

### 2.6 审计与协议服务层

审计由 `audit/logger.py` 的 `AuditLogger` 实现。每次请求都记录完整的五步思维链（上下文指代解析 → 意图风险筛查 → 工具选择 → 风险裁决 → 执行），并以哈希链方式落盘：每条事件的 `entry_hash` 由包含 `prev_hash` 的规范化 JSON 求 SHA-256 得到，`prev_hash` 指向上一条的 `entry_hash`。任何对历史事件的修改、删除或重排都会破坏链条，可由 `verify()` 检出。

代码印证：`src/safeops_agent/audit/logger.py` 的落盘逻辑把前一条哈希纳入当前事件再计算哈希。

```python
payload["prev_hash"] = self._last_entry_hash()
canonical = json.dumps(payload, ensure_ascii=False, sort_keys=True)
payload["entry_hash"] = hashlib.sha256(canonical.encode("utf-8")).hexdigest()
```

协议服务由 `mcp_stdio.py` 提供，实现符合 Model Context Protocol 的标准 JSON-RPC 2.0 stdio 服务端，支持 `initialize` 版本协商、`tools/list`、`tools/call` 和 `ping`。工具清单带 `inputSchema` 与安全注解，`tools/call` 内部复用同一套 `PolicyEngine` 裁决，使 CLI、Web 与 MCP 客户端共享一致的安全边界。

### 2.7 核心数据结构与模块映射

这些核心数据结构形成了本项目的内部契约。通过它们，系统可以把一次运维请求拆成多个可测试环节，而不是把意图识别、风险判断和执行混在同一个函数里。

| 数据结构 / 模块 | 责任 | 主要产出 |
| --- | --- | --- |
| `AgentResponse` | 封装一次请求的最终结果 | 结果摘要、风险等级、风险评分、思维链、确认令牌 |
| `PolicyDecision` | 记录一次策略裁决结论 | 是否允许、风险等级、原因、错误码、是否需确认 |
| `ToolSpec` | 声明一个白名单工具 | 名称、风险等级、参数 schema、处理函数、分类 |
| `ToolResult` | 封装一次工具执行结果 | 成功标志、结果摘要、结构化数据、错误信息 |
| `_ReasoningChain` | 累积可回放的五步思维链 | 逐步 stage/title/detail/inputs/outputs |
| `PendingActionStore` | 一次性确认令牌存储 | 精确保存待执行 (tool, args)，限时、绑定会话 |
| `AuditLogger` | 哈希链防篡改审计 | 落盘事件、`prev_hash`/`entry_hash`、`verify()` 校验 |

这种数据结构设计也便于后续扩展：新增工具只需注册一个 `ToolSpec`；新增安全规则只需扩展 `PolicyEngine`；新增意图理解通道只需实现 `LLMProvider` 抽象；替换审计后端也不影响上层决策逻辑。

## 3. 开发计划

### 3.1 阶段一：MVP 安全运维闭环

第一阶段目标是形成可运行的最小安全闭环：自然语言输入、规则意图识别、三级风险策略和工具执行。

验收标准：

- 能通过 CLI 发起自然语言运维请求；
- 能用规则匹配选出工具并抽取基础参数；
- 能区分低/中/高风险并作出放行/确认/拒绝处置；
- 能执行系统信息、资源指标、进程、端口等只读工具。

### 3.2 阶段二：双向护栏与哈希链审计

第二阶段目标是把"能执行"升级为"安全地执行、可追溯地执行"。系统需要在输入侧筛查高危意图、在参数侧拦截注入与敏感路径，并把每次决策拆成五步思维链、以哈希链方式落盘。

验收标准：

- 高危关键词与"危险动作 + 敏感路径"在意图阶段即被拦截；
- 参数注入字符、敏感路径、非法服务名/包名被拦截；
- 审计日志记录五步思维链；
- 审计事件形成哈希链，`--verify-audit` 可校验完整性。

### 3.3 阶段三：大模型意图理解与 MCP 协议

第三阶段目标是提升语义理解能力并接入标准协议，但不让大模型成为不受控的执行入口。系统需要支持大模型意图理解、失败回退、输出侧护栏，以及标准 MCP stdio 服务端。

验收标准：

- 接入 DeepSeek（OpenAI 兼容），支持超时与失败自动回退规则；
- 回退原因写入思维链，降级可见；
- 大模型追问/推理文本经输出侧护栏过滤后才展示与入库；
- 提供标准 JSON-RPC 2.0 MCP stdio 服务端，含版本协商。

### 3.4 阶段四：确认令牌、Web 工作台与提交收敛

第四阶段围绕安全性收尾与初赛交付展开。重点任务包括一次性确认令牌、Web 运维工作台、受管文件回滚，以及提交材料整理。

验收标准：

- 中风险操作签发一次性、限时、绑定会话的确认令牌；
- 确认阶段精确执行已预演动作，且确认后仍复核策略；
- Web 端可输入请求、查看工具清单、查看审计并一键确认中风险；
- README、操作手册、设计文档、演示脚本互相指向。

### 3.5 风险与回退计划

| 风险 | 回退方案 |
| --- | --- |
| 大模型 API 不可用或超时 | 8 秒超时后自动回退规则匹配，功能不中断 |
| 大模型输出越权或触碰高危操作 | 输出侧护栏过滤，命中高风险策略的文本整体屏蔽 |
| 无麒麟实机可验证 | Windows 返回兼容文本，整理 K01–K31 实机验证清单待补测 |
| 演示消耗真实 API 费用 | `SAFEOPS_LLM_DISABLED=1` 一键强制离线，测试全量离线 |

### 4.2 从规则匹配扩展为大模型意图理解

项目早期意图识别完全依赖关键词规则，虽然稳定，但对表达方式敏感。后来引入 DeepSeek 大模型意图理解，支持多轮上下文、指代解析和参数追问；同时保留规则通道作为回退。关键取舍是：大模型只增强"理解"，不触碰"执行权限"，其结果仍要经过完整的风险裁决。

### 4.3 从"确认即执行"收紧为一次性确认令牌

早期中风险确认使用 `--yes`，即重新理解意图后放行。但大模型意图理解存在非确定性，重跑不保证得到同一动作，存在"预演的是 A、确认后执行的是 B"的风险。项目为此引入一次性确认令牌：中风险裁决时精确保存 (tool, args) 并签发令牌，确认阶段凭令牌执行当初预演过的动作，令牌一次性、限时、绑定会话。

### 4.4 审计从流水日志升级为哈希链防篡改

早期审计只是 JSONL 流水日志，能记录但不能自证未被删改。项目将其升级为哈希链结构，并提供 `verify()` 校验：改动任意历史事件的内容、删除或重排中间事件，都会破坏链条并被检出。这把"报告是否可信"从"口头保证"变成了"可执行检查"。

## 5. 遇到的主要问题和解决方法

本章重点说明项目从"能运维"走向"安全运维、可审计运维"过程中遇到的关键问题。SafeOps Agent 面向的是操作系统运维执行，不是纯问答系统，因此问题集中在高危拦截、意图稳定性、大模型风险、确认一致性和审计可信几个方面。每个问题都对应了具体工程取舍：哪些指令能进入执行、哪些必须拦在门外、哪些必须交回给人确认。

### 5.1 高危自然语言指令需要在执行前拦截

问题表现：运维场景中，一句"删除根目录所有文件"或"格式化磁盘"如果被当成普通请求进入意图理解，一旦模型误选工具就可能造成不可逆后果。

解决方法：在意图理解之前设置输入侧护栏 `evaluate_intent`，对破坏性关键词与"危险动作 + 敏感路径"组合直接判为高风险拒绝，根本不进入工具选择。

代码印证：`src/safeops_agent/security/policy.py` 在护栏第一层即拦截高危关键词。

```python
for keyword in self.active_destructive_keywords:
    kw_compact = keyword.replace(" ", "").lower()
    ...
    return PolicyDecision(allowed=False, risk=RiskLevel.HIGH,
                          reason=f"命中高风险意图关键词：{keyword}",
                          error_code="INTENT_HIGH_RISK_KEYWORD")
```

实际效果：输入"删除根目录所有文件"时，系统直接返回"已拒绝执行：命中高风险意图关键词：删除根目录"，进程退出码非零，且不签发任何确认令牌。高危意图在最外层就被挡住，不消耗后续意图理解与执行资源。

### 5.2 大模型意图理解不稳定，需要可回退

问题表现：大模型意图理解语义能力强，但受网络、超时、额度和返回格式影响，可能失败或返回未知工具。如果主流程强依赖它，一次抖动就会中断运维。

解决方法：把大模型设计为可选增强通道，设置 8 秒超时；失败、返回未知工具或解析失败时自动回退规则匹配，并把回退原因写入思维链，使降级可见。

代码印证：`src/safeops_agent/llm/provider.py` 在调用异常时返回统一失败信号，由上层触发回退。

```python
try:
    result = self._chat_completion(messages)
except Exception:
    return None, {}, "LLM 调用失败", None
```

实际效果：无网络或无 API Key 时，系统自动使用规则通道完成意图识别，功能不中断；有网络时优先使用大模型理解，且回退过程在审计思维链中可见，而非静默发生。

### 5.3 大模型输出本身可能构成新的风险入口

问题表现：大模型的追问文本、推理文本会展示给运维人员并写入审计。如果不加约束，模型输出可能超长，甚至复述高危操作，形成新的风险入口。

解决方法：设置输出侧护栏 `_sanitize_llm_text`，对大模型追问/推理文本做长度截断，并让其经过与用户输入同一套高风险策略筛查，命中则整体屏蔽。

实际效果：大模型输出被纳入与用户输入相同的安全边界，形成"输入过滤 + 输出过滤"的双向护栏，避免模型成为绕过策略的旁路。

### 5.4 中风险操作"预演 A、执行 B"的不一致窗口

问题表现：中风险操作若采用"确认时重新理解意图再执行"，由于大模型意图理解存在非确定性，重跑可能得到不同的工具或参数，导致预演动作与实际执行动作不一致。

解决方法：引入一次性确认令牌。中风险裁决时精确保存 (tool, args) 并签发令牌，确认阶段凭令牌执行当初预演过的动作，不再重跑意图理解；令牌一次性、限时、绑定会话，且确认后仍复核策略。

代码印证：`src/safeops_agent/security/pending.py` 的 `consume` 保证令牌一次性、限时、绑定会话。

```python
record = actions.pop(action_id, None)      # 取出即删除，不可重放
...
if time.time() - float(record.get("created_at", 0)) > self.ttl:
    return None, f"确认令牌已过期（有效期 {self.ttl // 60} 分钟），请重新发起请求"
if record.get("session") != session:
    return None, "确认令牌与当前会话不匹配，已拒绝执行"
```

实际效果：中风险操作"重启 nginx 服务"会返回 dry-run 预演计划与一次性令牌，凭令牌执行的正是当初预演的动作；令牌用后即失效，伪造或过期令牌被拒绝。预演与执行严格一致的不一致窗口被消除。

### 5.5 审计日志可能被事后删改

问题表现：普通流水日志能记录决策，但无法证明记录本身没有被事后修改或删除，削弱了审计的可信度。

解决方法：把审计升级为哈希链结构，每条事件哈希包含前一条哈希；提供 `verify()` 校验内容篡改与链条断裂。

代码印证：`src/safeops_agent/audit/logger.py` 的 `verify()` 重算哈希并核对链条。

```python
recomputed = hashlib.sha256(
    json.dumps(event, ensure_ascii=False, sort_keys=True).encode("utf-8")
).hexdigest()
if recomputed != entry_hash:
    return {"ok": False, ..., "reason": "事件内容与哈希不符（内容被篡改）"}
if prev_hash is not None and event.get("prev_hash") != prev_hash:
    return {"ok": False, ..., "reason": "链条断裂（事件被删除或重排）"}
```

实际效果：`--verify-audit` 会返回 `ok`、已校验条数与首个异常行号。任何对历史事件的篡改、删除或重排都会被检出，使审计从"可记录"升级为"可自证完整"。

### 5.6 Windows 开发环境与麒麟目标环境的差异

问题表现：开发主要在 Windows 上进行，而 `systemctl`、`journalctl`、`ss`、`rpm` 等命令是麒麟/Linux 专有的，直接执行会报错或误导。

解决方法：工具在 Windows 上检测平台并返回兼容说明文本，在麒麟/Linux 上真实执行；同时整理麒麟实机验证清单，明确哪些结果是兼容输出、哪些需实机验证。

代码印证：`src/safeops_agent/tools/system.py` 按平台分支处理服务操作。

```python
if platform.system().lower() == "windows":
    return ToolResult(ok=True,
        summary=f"当前为 Windows 开发环境，将在麒麟/Linux 环境使用 systemctl {action} {service}")
ok, output = _run_command(["systemctl", action, service], timeout=15)
```

实际效果：开发与演示可在 Windows 上稳定进行，真实系统操作在麒麟/Linux 上落地；文档如实说明差异，避免夸大完成度。

### 5.7 演示与测试不应消耗真实 API 费用

问题表现：接入真实大模型后，如果测试和 CI 每次都调用在线模型，会产生费用，也会因网络抖动导致测试结果不确定。

解决方法：提供 `SAFEOPS_LLM_DISABLED=1` 环境变量，一键强制离线规则模式；测试与 CI 全量离线，保证确定性与零成本。

代码印证：`src/safeops_agent/llm/__init__.py` 的 `get_provider` 优先判断离线开关。

```python
if os.environ.get("SAFEOPS_LLM_DISABLED", "") == "1":
    return RuleBasedProvider()
```

实际效果：130+ 个单元测试全部离线确定性通过，不消耗任何 API 费用；需要演示大模型时再启用在线通道，两种模式一键切换。

## 6. 作品特征描述

本章从评审角度总结 SafeOps Agent 的作品特征。与直接把大模型接到 shell 的方案相比，本作品的重点在于安全护栏优先、最小权限和可审计；与纯问答式助手相比，本作品的重点在于真实的操作系统运维能力与三级风险处置。

### 6.1 安全护栏优先

系统把安全护栏放在意图理解之前和工具执行之前，形成"输入过滤 → 意图理解 → 参数校验 → 风险分级 → 确认令牌 → 执行 → 哈希链审计"的完整闭环。任何一步不通过，流程即终止。这使得"能不能做"始终由确定性策略决定，而不是由大模型自由发挥。

### 6.2 三级风险与一次性确认令牌

系统把操作按风险分为低/中/高三级差异化处置：低风险只读操作直接执行，高风险破坏性操作默认拒绝，中风险变更操作签发一次性、限时、绑定会话的确认令牌。确认阶段精确执行已预演动作并再次复核策略，杜绝"预演 A、执行 B"和令牌重放。

### 6.3 最小权限白名单工具

系统不提供"执行任意命令"的能力，所有运维动作都封装成带风险标注的白名单工具，工具之外无执行路径。参数经过注入字符、敏感路径和安全标识符校验；受管文件写入被限制在项目内工作区，写前快照、可真实回滚。

### 6.4 五步思维链与哈希链审计

每次请求都记录上下文指代解析、意图风险筛查、工具选择、风险裁决、执行五步可回放思维链，并以哈希链方式落盘。审计不仅"可记录"，而且"可自证完整"：`--verify-audit` 能检出任意内容篡改与链条断裂。这让每一次决策都可解释、可追溯、可校验。

### 6.5 大模型可控增强与离线回退

大模型在本项目中只承担意图理解增强，不作为执行入口。系统通过输入护栏、输出护栏、8 秒超时、失败回退和离线开关控制大模型风险；无网络、无 Key 或强制离线时，规则通道保证功能不中断。这种边界让系统在不开启大模型时仍具备完整安全运维能力。

### 6.6 标准 MCP 协议与双入口交付

系统实现符合 Model Context Protocol 的标准 JSON-RPC 2.0 stdio 服务端，工具带 inputSchema 与安全注解，可被通用 MCP 客户端复用；同时提供 CLI 与 Web 双入口。三个入口共享同一套安全策略与审计逻辑，保证无论从哪个入口进入，安全边界都一致。

## 9. 比赛收获

### 9.1 对安全智能运维的理解

通过项目建设，我们更清楚地认识到，智能运维系统的价值不只在"听懂自然语言"，更在"安全地执行"和"可信地记录"。把大模型直接接到 shell 看似能力最强，实则风险不可控；真正可交付的方案，是把大模型约束在意图理解层，把执行权牢牢握在确定性策略手里。

### 9.2 对智能体工程边界的理解

本项目让我们认识到，大模型适合理解意图和组织语言，但不适合直接决定"能不能执行"。面向运维执行场景，智能体必须有明确的输入护栏、输出护栏、风险分级、确认机制和失败回退。因此 SafeOps Agent 把确定性安全策略放在前面，把大模型放在可控增强层，通过双向护栏和一次性确认令牌把风险限制在可解释、可回退的范围内。

### 9.3 对工程交付的理解

比赛提交不是只提交代码，还需要 README、设计文档、操作手册、演示视频和可复现的测试形成闭环。项目在建设过程中坚持离线可复现、审计可自证、文档与实现一致，使评委能够从多个入口理解作品，并亲手验证安全链路。

### 9.4 对后续改进方向的认识

当前系统已经能支撑初赛演示和安全运维闭环，但仍有提升空间。后续可以在真实麒麟系统上执行 K01–K31 验证清单，落地专用低权用户 + sudo 白名单执行器；扩展日志根因分析、资源阈值告警和受控故障自愈；并把审计校验、护栏拦截等接入更完整的 CI 流程，进一步增强作品的工程完备度。
